# Crush Development Guide

## Project Overview

Crush is a terminal-first AI coding assistant written in Go. The module path is
`github.com/charmbracelet/crush`.

The executable starts in `main.go`, which optionally enables pprof via
`CRUSH_PROFILE` and then hands off to the Cobra/Fang CLI in
`internal/cmd/root.go`.

## Repository Structure

```text
main.go                     Program entrypoint.
Taskfile.yaml               Canonical build/test/lint/dev tasks.
crush.json                  Repo-local Crush config; configures gopls.
sqlc.yaml                   sqlc generation config for SQLite queries.
internal/
  app/                      Application wiring, lifecycle, background services.
  agent/                    Agent orchestration, prompts, tool registry, MCP.
  cmd/                      CLI commands and setup.
  config/                   Config loading, provider resolution, defaults.
  db/                       sqlc-generated DB layer, migrations, SQL queries.
  filetracker/              Tracks files touched per session.
  history/                  Prompt/file history services.
  lsp/                      LSP manager and helpers.
  message/                  Message/content model used by agent + persistence.
  permission/               Tool permission checks and allowlists.
  pubsub/                   Internal event bus.
  session/                  Session CRUD and todo persistence.
  shell/                    Shell execution and background job management.
  skills/                   Agent skill discovery/loading.
  ui/                       Bubble Tea TUI. Read `internal/ui/AGENTS.md` first.
```

Notable generated areas:

- `internal/db/*.sql.go`, `internal/db/db.go`, `internal/db/models.go`,
  `internal/db/querier.go` are generated by `sqlc` and marked `DO NOT EDIT`.
- JSON schema output is generated into `schema.json` by the `schema` task.
- Hyper provider data is generated via `go generate ./internal/agent/hyper/...`.

## Build, Run, Test, and Maintenance Commands

These commands are defined in `Taskfile.yaml` unless noted otherwise.

### Core commands

- Build: `task build`
  - Uses `CGO_ENABLED=0` and `GOEXPERIMENT=greenteagc`.
  - Injects version info via ldflags when `git describe --long` succeeds.
- Run built binary: `task run -- <args>`
- Run with local Catwalk: `task run:catwalk -- <args>`
- Dev run with profiling enabled: `task dev`
- Install locally: `task install`
- Direct Go alternatives:
  - `go build .`
  - `go run .`
  - `go install .`

### Tests

- Full test suite: `task test`
  - Runs `go test -race -failfast ./...`
- Direct Go alternative: `go test ./...`
- Re-record agent VCR fixtures: `task test:record`
  - Deletes `internal/agent/testdata` first, then runs the agent tests again.
- Update golden files: `go test ./... -update`
  - The README mentions package-scoped updates for snapshot/golden tests.

### Lint and formatting

- Lint: `task lint`
  - Runs `task lint:log`
  - Runs `golangci-lint run --path-mode=abs --config=.golangci.yml --timeout=5m`
  - Sets `GOEXPERIMENT: null` for linting.
- Auto-fix lint issues: `task lint:fix`
- Log-message capitalization check only: `task lint:log`
- Format Go: `task fmt`
  - Runs `gofumpt -w .`
- Format web assets: `task fmt:html`
  - Formats `internal/cmd/stats/index.html`, `.css`, and `.js` with Prettier.
- Modernize Go code: `task modernize`

### Generation and profiling

- Generate config schema: `task schema`
- Refresh Hyper embedded provider data: `task hyper`
- CPU profile UI: `task profile:cpu`
- Heap profile UI: `task profile:heap`
- Allocation profile UI: `task profile:allocs`

### Release-related tasks

- `task release` exists and performs tagging/pushing after branch/cleanliness/CI
  preconditions. Do not run it casually.

## Go and Tooling Baseline

- `go.mod` declares **Go 1.26.0**.
- The project uses `cobra` plus `fang` for the CLI.
- The TUI uses Bubble Tea v2, Lip Gloss v2, and related Charm libraries.
- LLM/provider abstraction is primarily through `charm.land/fantasy`.
- Provider metadata is sourced from Catwalk (`charm.land/catwalk`).
- SQLite persistence is generated through `sqlc` from:
  - schema: `internal/db/migrations`
  - queries: `internal/db/sql`
  - output: `internal/db`

## Code Organization and Architecture Notes

### App wiring

- `internal/app/app.go` constructs the app service graph.
- It wires sessions, messages, history, permissions, file tracking, LSP, MCP,
  update checks, and the agent coordinator.
- Shutdown is coordinated and parallelized; it cancels agents first, then kills
  background shells, LSP clients, and cleanup hooks.

### CLI behavior

- `internal/cmd/root.go` is the main CLI entry.
- `setupApp()` resolves cwd, loads config, creates the data dir, registers the
  project, connects the DB, and initializes metrics.
- `.crush` data dir creation also writes a `.gitignore` containing `*` if one is
  absent.
- Non-interactive input can be prefixed from stdin via `MaybePrependStdin`.

### Agent system

- `internal/agent/agent.go` is the core session agent implementation.
- Session runs are queued per session when a request is already active.
- MCP server instructions are appended into the system prompt dynamically.
- Anthropic/Bedrock/Vercel cache-control options are applied unless
  `CRUSH_DISABLE_ANTHROPIC_CACHE` is truthy.
- Auto-summarization is triggered based on context-window thresholds.
- When summaries exist, `getSessionMessages()` trims history to start at the
  summary message and reinterprets that message as a user message.
- Tool-result media is rewritten into user attachments for providers that do not
  support media in tool results.

### Message model

- `internal/message/content.go` stores messages as a slice of typed content
  parts (`TextContent`, `ReasoningContent`, `ToolCall`, `ToolResult`, etc.).
- Messages are incrementally updated during streaming by mutating/replacing
  parts.
- Text attachments are embedded into prompt text using `<file path='...'>`
  wrappers.
- Non-text attachments become `fantasy.FilePart` values.

### Sessions and todos

- `internal/session/session.go` persists todo lists on sessions as JSON.
- Todo items include `content`, `status`, and `active_form`.
- Session IDs for agent-tool child sessions use the format
  `messageID$$toolCallID`.

### Providers and provider cache

- `internal/config/provider.go` loads provider metadata from Catwalk and Hyper.
- Provider data is cached under:
  - `$XDG_DATA_HOME/crush/*.json` when `XDG_DATA_HOME` is set
  - `%LOCALAPPDATA%/crush/*.json` on Windows
  - `$HOME/.local/share/crush/*.json` otherwise
- `Providers()` uses `sync.Once`, so tests that depend on provider state must
  reset global provider state.
- Errors fall back toward cached or embedded provider data rather than failing
  hard immediately.

## UI-Specific Rules

Read `internal/ui/AGENTS.md` before changing TUI code. Key observed guidance:

- Keep expensive work and IO out of `Update`; use `tea.Cmd`.
- Prefer direct state mutation/messages over command indirection when possible.
- Keep logic in the main model (`internal/ui/model/ui.go`).
- Components should be “dumb”: expose methods, render themselves, avoid owning
  Bubble Tea message handling.
- Use `github.com/charmbracelet/x/ansi` for ANSI-aware string manipulation.
- Use styles from `styles/styles.go`; prefer semantic style fields over
  hardcoded colors.
- Account for padding/borders in width calculations.
- Use `tea.Batch()` for multiple commands.

## Naming, Style, and Conventions

Observed/repo-documented conventions:

- Format with `gofumpt`; imports are managed with `goimports`.
- Standard Go naming: exported PascalCase, unexported camelCase.
- Prefer explicit error wrapping with `fmt.Errorf(... %w)`.
- Pass `context.Context` as the first parameter for operations that need it.
- JSON tags use `snake_case`.
- File permissions use octal notation such as `0o755` and `0o644`.
- Log messages must start with a capital letter.
- Standalone comments should start with a capital letter and end with periods.
- Struct embedding is used for composition.
- Small consumer-owned interfaces are preferred.

## Testing Patterns

Observed from existing tests:

- Use `testing` plus `github.com/stretchr/testify/require`.
- Many tests call `t.Parallel()`.
- Use `t.TempDir()` for filesystem isolation.
- Use `t.Setenv()` to isolate environment-dependent behavior.
- Provider/config tests often create fresh syncers and reset global provider
  state because provider loading is cached with `sync.Once`.
- Some tests rely on embedded/cached provider fallbacks instead of live network
  access.
- There are snapshot/golden-style tests in the repo; update them with
  `go test ./... -update` when intended output changes.
- Agent tests may use recorded fixtures under `internal/agent/testdata`.

The existing guide in this repo also documents a mock-provider pattern:

```go
originalUseMock := config.UseMockProviders
config.UseMockProviders = true
defer func() {
    config.UseMockProviders = originalUseMock
    config.ResetProviders()
}()

config.ResetProviders()
```

Use that pattern when writing tests that should not hit real providers.

## Gotchas and Non-Obvious Behaviors

- `internal/db` contains generated files; edit SQL/migrations instead of the
  generated Go where applicable.
- Provider loading is memoized with `sync.Once`; reset provider globals in tests
  if you need a fresh state.
- `task lint` and `task lint:fix` temporarily set `GOEXPERIMENT` to `null`,
  while normal build/test tasks use `greenteagc`.
- Non-interactive mode auto-approves permissions for the created session.
- The app waits for MCP initialization before reading MCP tools for a run.
- Message history intentionally injects a reminder about the todo-list tool when
  preparing prompts for non-sub-agents.
- Tool-result media handling differs by provider; do not assume all providers
  accept image/media tool results.
- Logs for the running app are documented in `README.md` as
  `./.crush/logs/crush.log` relative to the project.

## Safe Change Guidance for Future Agents

- Read `internal/ui/AGENTS.md` before touching anything under `internal/ui/`.
- Prefer `task build`, `task test`, `task lint`, and `task fmt` over ad hoc
  commands; those tasks encode repo-specific env and flags.
- When changing DB queries or schema, inspect `sqlc.yaml` and regenerate related
  artifacts rather than hand-editing generated files.
- When changing provider/config behavior, check both runtime code in
  `internal/config/` and tests that reset provider global state.
- When changing agent execution flow, review interactions among
  `internal/agent/`, `internal/message/`, `internal/session/`, and
  `internal/permission/` because the behavior is tightly coupled.
- When changing CLI/TUI startup, follow the flow from `main.go` to
  `internal/cmd/root.go` to `internal/app/app.go`.
